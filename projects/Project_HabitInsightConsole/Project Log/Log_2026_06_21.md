## 1. Work Summary

Today, I implemented the core class structure for `Habit Insight Console` and started building the basic console workflow.

The main focus was separating class responsibilities clearly. The `Habit` class was designed as an abstract parent class that manages shared habit information and common behavior. `StudyHabit`, `ExerciseHabit`, and `GeneralHabit` were implemented as child classes that manage type-specific habit details.

I also created the main supporting classes: `HabitTracker` for managing multiple habit objects, `HabitStatistics` for analyzing habit data, and `HabitInsightConsole` for running the console-based program.

At the current stage, the main console program has implemented the **Habit Management menu and the Add Habit feature**.

## 2. Completed Work

The following classes were written or structured today:

| Class | Role |
|---|---|
| `Habit` | Abstract parent class that contains shared habit information, completion rate logic, streak tracking, status management, and progress summary behavior. |
| `StudyHabit` | Child class for study-related habits. It manages study-specific information such as subject and study session data. |
| `ExerciseHabit` | Child class for exercise-related habits. It manages exercise-specific information such as exercise type and workout data. |
| `GeneralHabit` | Child class for general habits. It manages simple memo or note-based habit information. |
| `HabitStatus` | Enum used to represent habit status, including `ACTIVE`, `PAUSED`, and `ARCHIVED`. |
| `HabitCategory` | Enum used to categorize habits for analysis, including study, exercise, health, productivity, personal, and general categories. |
| `HabitTracker` | Management class responsible for storing, adding, removing, searching, and printing habit objects. |
| `HabitStatistics` | Analysis class responsible for calculating summary information such as completion rate, streak results, and category summaries. |
| `HabitInsightConsole` | Main execution class responsible for displaying menus, receiving user input, and connecting the tracker/statistics workflow. |

## 4. Issues / Bugs Found

### 1. Habit Management menu loop needs improvement

When the user enters an invalid number in the Habit Management menu, the program should return to the Habit Management menu instead of going back to the main menu.

Target behavior:

```text
Display Habit Management menu
→ Receive user input
→ If input is invalid, print "Invalid choice"
→ Display Habit Management menu again
```

To implement this properly, the Habit Management menu display and input handling should be placed inside the `while` loop.

### 2. switch-case `break` statements need to be checked

Some `switch` statements may continue into the next case if `break` is missing.

After handling Add Habit, the case should explicitly end with:

```java
case 1:
    // add habit logic
    break;
```

### 3. Scanner input handling needs continued checking

When using `nextLine()` after `nextInt()`, the input buffer may cause issues. Areas where numeric input is followed by string input need to be reviewed carefully.

Example:

```java
int choice = scanner.nextInt();
scanner.nextLine();
String title = scanner.nextLine();
```

### 4. Streak calculation needs testing

The `markCompleted()` logic should be tested with sample data to confirm that streaks are calculated correctly.

Cases to test:

```text
First completion
Duplicate completion on the same day
Completion after yesterday
Completion after skipping several days
```

### 5. HabitStatistics output needs validation

The `HabitStatistics` summary report needs to be tested to confirm that the calculated values match the intended behavior.

Items to verify:

```text
Overall completion rate
Best streak habit
Highest completion rate habit
Lowest completion rate habit
Number of habits by category
```

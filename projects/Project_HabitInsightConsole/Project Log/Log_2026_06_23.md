## 4. Current Functional Flow

The current application now works with this improved lifecycle:

```text
Program Start
→ Load habits from Habit.csv
→ Store loaded Habit objects in ArrayList<Habit>
→ Pass habits list to HabitTracker and HabitStatistics
→ Run console menu
→ Add / Search / Remove / Manage / Complete / Report
→ Exit Program
→ Save current habits list back to Habit.csv
```

This is an important architectural improvement because the application no longer loses all data when the program ends.

The current persistence model is:

```text
CSV File
→ HabitFileStorage.loadHabits()
→ ArrayList<Habit>
→ HabitTracker
→ User modifies habits through console
→ HabitFileStorage.saveHabits()
→ CSV File
```

This structure also prepares the project for future database-backed development, because it introduces the idea of loading data into application memory, modifying it through a service-like class, and saving the final state back to storage.

## 5. Design Decisions Made

### 1. Keep `ArrayList` as the runtime data structure

I decided not to replace `ArrayList` with direct file access. Instead, the file is used only for persistence.

```text
Runtime data management: ArrayList<Habit>
Long-term storage: Habit.csv
```

This keeps the application simple and allows `HabitTracker` to continue managing habit objects during execution.

### 2. Add a separate `HabitFileStorage` class

File logic was separated into its own class instead of being placed inside `HabitTracker`.

This keeps responsibilities clearer:

```text
HabitInsightConsole
- Console menu flow
- User input
- Program start / exit workflow

HabitTracker
- Manage Habit objects
- Add, remove, search, complete

HabitFileStorage
- Save habits to file
- Load habits from file
- Convert Habit objects to CSV lines
- Convert CSV lines back into Habit objects
```

This was a good responsibility separation decision.

### 3. Save the full list on exit

Instead of appending only new habits to the CSV file, the program saves the full current habit list on exit.

This is simpler and safer for now because it correctly reflects removed or modified habits.

```text
Current ArrayList state
→ overwrite Habit.csv
→ latest state preserved
```

## 6. Issues / Bugs Found

### 1. `StudyHabit` was not loading correctly

A major bug appeared after file persistence was added.

Observed behavior:

```text
ExerciseHabit loads correctly
GeneralHabit loads correctly
StudyHabit is skipped during loading
After exiting again, StudyHabit disappears from the CSV file
```

Root cause:

```text
The saved habit type string and the loading switch-case string did not match.
```

The loader expected:

```text
STUDY
```

but the saved line previously used a different format such as:

```text
Study
```

Because Java string comparison is case-sensitive, the Study line went into the `default` branch and returned `null`. As a result, the Study habit was not added to the loaded `ArrayList`.

Then, when the program exited, the application saved the current list again without the skipped Study habit. This made it look like the Study habit was deleted.

Fix:

```text
Use a consistent habit type identifier:
STUDY
EXERCISE
GENERAL
```

### 2. CSV parsing still needs `trim()` protection

The current parser can still be sensitive to extra spaces.

Recommended improvement:

```java
String habitType = parts[0].trim().toUpperCase();
String title = parts[1].trim();
HabitCategory category = HabitCategory.valueOf(parts[2].trim());
int targetCount = Integer.parseInt(parts[3].trim());
int completedCount = Integer.parseInt(parts[4].trim());
String specificField = parts[5].trim();
```

This will make file loading more stable.

### 3. Completion count restoration may not be fully accurate

The current loading logic restores `completedCount` by repeatedly calling `markComplete()`.

This may become inaccurate because `markComplete()` contains date-based duplicate completion prevention. If `completedCount` is greater than 1, repeated calls during loading may not restore the exact original count and streak state.

This is not the highest priority right now, but it should be improved later by adding a more direct restoration method or by saving more fields such as:

```text
completedCount
currentStreak
bestStreak
lastCompletedDate
status
```

### 4. Scanner usage still needs cleanup

Some helper methods still create or depend on input handling in a way that can make `nextInt()` / `nextLine()` behavior fragile.

Future cleanup should standardize this rule:

```text
Create Scanner once in main
Pass the same Scanner into helper methods
Handle newline buffer carefully after nextInt()
```

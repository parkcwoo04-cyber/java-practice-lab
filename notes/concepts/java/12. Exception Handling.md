
An `exception` is a situation in which a program behaves differently from what the programmer expected. Without `exception handling`, the program usually stops execution when an `exception` occurs. `exception handling` enables the program to continue running in a stable manner instead of terminating abruptly.

## Types of Program Error

### Compile Error

A `compile error` is an error caused by syntax or structural issues in the code. Since the program cannot be compiled, these errors must be corrected before the program can run.

```java
int i = "Hello"; // wrong variable type
```

### Runtime Error

`runtime error` occurs during program execution after the code has been successfully compiled. Although there are no syntax errors, flaws in the program logic or design can cause runtime errors.

```java
public class _01_TryCatch {  
    public static void main(String[] args) {  
        int[] array = new int[3];  
        array[5] = 1;  
        System.out.println(array[5]);  
    }  
}
```
```
Outcome

Exception in thread "main" java.lang.ArrayIndexOutOfBoundsException: Index 5 out of bounds for length 3
```

```java
public class _01_TryCatch {  
    public static void main(String[] args) {  
        S s = new S();  
        s.methodA();  
    }  
  
}  
  
class S {  
    public void methodA() {  
        this.methodA();  
    }  
}
```
```
Outcome

Exception in thread "main" java.lang.StackOverflowError
	at chap_13.S.methodA(_01_TryCatch.java:13)
	at chap_13.S.methodA(_01_TryCatch.java:13)
	at chap_13.S.methodA(_01_TryCatch.java:13)
```

Although both examples are runtime errors, they represents different types of problems.

The first case involves an `exception` that can be handled through code. For example, attempting to read a file does not exist or accessing an array beyond its valid index range can cause exceptions. These situations can be anticipated and handled using exception handling.

In contrast, the second case involves a method repeatedly calling itself. As the method continues to invoke itself indefinitely, the stack memory used to store method call information eventually becomes full, resulting in a `StackOverFlowError`. 

This is a serious error that cannot be recovered from through normal exception handling. Rather than trying to handle it in code, it should be prevented during the software design and implementation stages.

## try-catch

To prevent a program from stopping unexpectedly when an exception occurs, Java provides a feature called `try-catch`.

```java
try {
	// code that may throw an exception
} catch (IOException e) { // exception type
	// exception-handling code
}
```

- `try {...}`: It is the section where the program attempts to execute code that may cause an exception. Any code that could potentially throw an exception is placed inside this block.

- `catch (IOException e) {...}`: It is the section that handles exceptions. When an exception occurs in the `try` block, the normal flow of execution stops immediately, and control is transferred to the `catch` block. The exception handling code is than executed.

```java
try {  
    System.out.println("Start dividing");  
    int result = 3/0;  
    System.out.println("3/0 = " + result);  
} catch (Exception e) {  
    System.out.println("Error Occurred");  
    System.out.println("Exception: " + e.getMessage());  
}  
System.out.println("End dividing");
```
```
Outcome

Start dividing
Error Occurred
Exception: / by zero
End dividing
```

When `3/0` is executed inside the `try` block, an exception is thrown. The normal flow of execution is interrupted, and control is transferred to the `catch` block. The exception-handling code is then executed, allowing the program to continue running and finish normally.

## Various Exception Scenarios

### ArrayIndexOutOfBoundsException

This exception occurs when an attempt is made to access an element outside the valid range of an array.

```java
int[] array = new int[3];
arr[5] = 100;
```

In this code, the array is created with a size of three elements. However, the program attempts to access index `5`, which is beyond the valid range of the array. As a result, an **ArrayIndexOutOfBoundsException** is thrown.

### ClassCastException

This exception occurs when attempting to cast an object to a type that is not compatible with its actual runtime type.

```java
Object obj = "test";
int numb = (int) obj;
```

In this example, `obj` actually references a `String` object. However, the code attempts to cast it to an `Integer`, which causes a `ClassCastException`.

## Multiple Catch Blocks

Java allows the use of multiple `catch` blocks to handle different types of exceptions.

```java
try {  
    String text = "abc";  
    int number = Integer.parseInt(text);  
  
    int[] array = new int[3];  
    array[5] = 100;  
} catch (NumberFormatException e) {  
    System.out.println("Invalid number format.");  
} catch (ArrayIndexOutOfBoundsException e) {  
    System.out.println("Array index is out of bounds.");  
} catch (Exception e) {  
    System.out.println("An unexpected exception occurred.");  
}  
  
System.out.println("Program finished.");
```
```
Outcome

Invalid number format.
Program finished.
```

This code can potentially throw two different types of exceptions. A `NumberFormatException` may occur when converting a string to an integer, and an `ArrayIndexOutOfBoundsException` may occur when accessing an array index outside its valid range.  

When an exception is thrown, the normal flow of execution immediately stops and control is transferred to the matching `catch` block. As a result, the remaining statements in the `try` block are not executed. Therefore, only one exception is handled during a single execution of the program.

Each `catch` block displays a message corresponding to the exception type it handles. The final `catch (Exception e)` block serves as a general handler for exceptions that are not caught by the preceding `catch` blocks. In this example, the thrown exception is already handled by one of the earlier `catch` blocks, so the final `catch` block is not executed.

By placing multiple `catch` blocks after a single `try` block, a program can respond differently to different types of exceptions.

The order of `catch` blocks is very important. In the previous example, the exception thrown in the `try` block is also a subtype of `Exception`. However, it is already handled by the `catch (NumberFormatException e)` block, so the `catch (Exception e)` block is never reached.

Since `Exception` is the superclass of most exception types, a `catch (Exception e)` block can handle a wide range of exceptions. For this reason, it should always be placed after more specific exception handlers.

If a `catch (Exception e)` block is placed before a more specific `catch` block, the compiler will report an error because the more specific exception can never be reached.

## Controlling Program Flow by Throwing Exceptions

The `throw` statement allows a programmer to create and throw an exception explicitly.

When an exception is thrown, normal program execution stops and control is transferred to the appropriate `catch` block.

Intentional exception throwing is commonly used to validate input, enforce business rules, and prevent invalid program states.

```java
int age = 17;
try {
	if(age < 19) {
		System.out.println("Invalid Age");
	} else {
		System.out.println("Transaction Successful");
	}
} catch (Exception e) {
	e.printStackTrace();
}
```

While an `if` statement is sufficient for handling this situation, the condition may sometimes be considered a critical business rule violation. In such cases, a programmer can explicitly throw an exception to interrupt the normal flow of execution and trigger appropriate error-handling procedures, such as notifying an administrator or terminating the operation.

```java
int age = 17;
try {
	if(age < 19) {
		throw new Exception("Invalid Age");
	} else {
		System.out.println("Transaction Successful");
	}
} catch (Exception e) {
	e.printStackTrace();
}

```

## finally

The `finally` block is used for operations that must always be performed. It executes regardless of whether the `try` block finishes successfully or and exception occurs and is handled by a `catch` block.

```java
try {  
    System.out.println("Enter the restaurant");  
    throw new Exception("Restaurant Closed");  
} catch (Exception ex) {  
    System.out.println(ex.getMessage());  
} finally {  
     System.out.println("Exit the restaurant");  
}
```

In this example, the program first prints the message `Enter the restaurant`. It then explicitly throws an exception with the message `Restaurant Closed`. 

Once the exception is thrown, execution immediately leaves the `try` block and moves to the `catch` block. The `catch` block prints the exception message, and the `finally` block is executed afterward.

In this case, the `try`, `catch`, and `finally` blocks are all executed.

However, if no exception had been thrown and the `try` block had completed normally, the `cach` block would not have been executed. The `finally` block, on the other hand, would still execute regardless of whether an exception occurred.

The `try`, `catch`, and `finally` blocks can be combined in different ways.

Depending on the situation, you can choose the structure that best fits your exception-handling needs.

## Try with resources

`finally` block is an important feature when there is code that must always be executed, such as resource cleanup operations.

However, manually releasing resources in a `finally` block can be repetitive and prone to mistakes. To simplify this process, Java provides a feature called `try-with-resources`, which automatically closes resources when they are no longer needed.

The following two examples illustrate the difference between these approaches.

```java
MyFileWriter writer = null;  
try {  
    writer = new MyFileWriter();  
    writer.write("I want IceCream");  
} catch (Exception e) {  
    e.printStackTrace();  
} finally {  
    try {  
        writer.close();  
    } catch (Exception e) {  
        throw new RuntimeException(e);  
    }  
}
```
```
Outcome


Adding context to the file..
Context: I want IceCream
Closing file
```

```java
try (MyFileWriter writer = new MyFileWriter()) {  
    writer.write("I want IceCream");  
} catch (Exception e) {  
    e.printStackTrace();  
}
```
```
Outcome

Adding context to the file..
Context: I want IceCream
Closing file
```

Both examples produce the same output, but their implementations differ significantly. Compared to using a `finally` block, `try-with-resources` greatly simplifies the code and reduces boilerplate.

## Custom Exception

Up to this point, we have dealt with exceptions provided by the Java platform. However, Java also allows developers to define their own exception types, known as `custom exceptions`, to represent application-specific conditions.

Before a custom exception can be used, a new exception class must be created by the developer.

```java
public class AgeLessThan19Exception extends Exception {  
    public AgeLessThan19Exception(String message) {  
        super(message);  
    }  
}
```

A class named `AgeLessThan19Exception` has been created. This class extends the `Exception` class and passes an error message to the parent exception class using `super(message)`.

```java
int age = 17;  
try {  
    if (age < 19) {  
        throw new AgeLessThan19Exception("Age is less than 19");  
    } else {  
        System.out.println("Valid Age");  
    }  
} catch (AgeLessThan19Exception e) {  
    System.out.println(e.getMessage());  
}
```
```
Outcome

Age is less than 19
```

Looking at the execution flow, the value of `age` is less than 19, so an `AgeLessThan19Exception` is thrown and handled by the `catch` block.

By using custom exceptions, the intent of the code becomes clearer, and exception-handling logic can be separated according to thy type of exception.

## Delegating Responsibility for Exception Handling

In most cases, a `try-catch` statement handles exceptions directly within the code block where they occur. However, by using the `throws` keyword, a method can pass the responsibility for handling an exception to its caller.

This approach is useful in large-scale applications where exceptions from multiple methods need to be managed and handled in a centralized location.

```java
public static void main(String[] args) {  
    System.out.println("Calling writeFile() ...");  
    try {  
        writeFile();  
    } catch (IOException e) {  
        System.out.println(e.getMessage());  
    }  
    System.out.println("Program has been successfully closed.");  
}  
  
public static void writeFile() throws IOException {  
    FileWriter writer = new FileWriter("test.txt");  
    throw new IOException("Failed to write file");  
}
```
```
Outcome

Calling writeFile() ...
Failed to write file
Program has been successfully closed.
```

In this example, the exception is not handled inside the `writeFile()` method. Instead, the method declares `throws IOException`, which passes the responsibility for handling the exception to its caller.

As a result, `writeFile()` focuses only on file-writing operations, while the `main()` method handles the exception using a `try-catch` block.

This separation of responsibilities improves code organization and makes exception handling easier to manage, especially in large applications.
